Event Sourcing / Audit Logs

Centralized, Chronological Record Keeping
EVENT (CHANGE) A record of a state change (e.g., "Status changed from Pending to Approved").
↓
APPEND-ONLY DATABASE A log-structured table (Cassandra, Kafka, specialized storage).
Log Entry: ID + Time + Data (Protected by file/database permissions)
↓
AUDIT TRAIL A sequence of records, reconstructed chronologically. Proves: The sequence of changes over time.
↓
CORE LIMITATION: TRUST IN CENTRAL ADMIN The log file can be secretly edited or replaced by a malicious administrator with root access. Integrity is not mathematically enforced.

NOVAK Architecture

Cryptographically Linked & Verifiable Process
THE PROCESS (NIPS + NOVAK) Rule + Data + Previous Hash + Time.
↓
REVELATION CHAINING SHA-256 uses the **Previous Hash** as a required input for the current receipt.
HASH (PREV_HASH + EVENT) → NEW HASH (Integrity is mathematically enforced)
↓
REVELATION HASH The new hash is the immutable link. Proves: The entire sequence is intact and nothing has been deleted or inserted.
↓
CORE STRENGTH: CRYPTOGRAPHIC IMMUTABILITY Changing a single historical event breaks the subsequent chain of hashes, providing instant, mathematical proof of tampering.
